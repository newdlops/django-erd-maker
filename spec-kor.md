# Django ERD Maker VS Code Extension 명세

## 1. 개요

이 프로젝트는 Django 프로젝트를 분석해서 인터랙티브한 ERD를 생성하고, 이를 VS Code 웹뷰 안에서 보여주는 확장이다.

확장은 다음을 만족해야 한다.

- 대상 워크스페이스에서 Django 앱과 모델을 찾을 것
- Rust 기반 엔진으로 모델을 정적 분석할 것
- 프로젝트 전체의 관계 그래프를 구성할 것
- 벡터 기반 ERD로 렌더링할 것
- 대규모 스키마에서도 사용할 수 있는 상호작용과 레이아웃 제어를 제공할 것
- 장기적으로 확장하기 쉬운 코드 구조를 유지할 것

## 2. 제품 목표

주요 목표는 Django 애플리케이션을 실행하지 않고도, 개발자가 VS Code 안에서 프로젝트 구조를 빠르게 이해할 수 있도록 ERD를 시각화하는 것이다.

## 3. 범위

### 포함 범위

- VS Code 확장 UI와 명령
- Django 모델 탐색과 정적 분석
- 프로젝트 전체 관계 그래프 구성
- 웹뷰 기반 인터랙티브 ERD 렌더링
- 복수의 레이아웃 전략
- 향후 기능 추가를 고려한 확장 가능한 구조

### 초기 명세에서 제외

- 실제 데이터베이스 introspection
- migration 그래프 시각화
- Django 런타임 부팅을 강제 전제로 하는 방식
- Django 이외 ORM 지원

## 4. 핵심 기능 요구사항

### 4.1 Django 프로젝트 분석

- 확장은 현재 워크스페이스에서 Django 앱과 모델 정의를 스캔해야 한다.
- 분석은 가능한 한 프로젝트 코드를 실행하지 않는 정적 분석 중심이어야 한다.
- 분석 엔진은 속도와 예측 가능한 성능을 위해 Rust로 구현해야 한다.
- 확장 측에서는 분석기를 분리 가능한 서브시스템으로 다뤄서, 이후 분석 엔진이 독립적으로 발전할 수 있어야 한다.

### 4.2 모델 추출

- 분석기는 프로젝트 앱에 정의된 Django 모델을 식별해야 한다.
- ERD 렌더링에 필요한 메타데이터를 추출해야 한다.
  - 앱 이름
  - 모델 이름
  - 필드
  - `@property`가 붙은 계산 속성
  - 사용자 정의 모델 메서드
  - 필드 타입
  - 선택 가능한 옵션을 선언한 필드의 choice 또는 enum 메타데이터
  - 관계 타입
  - 대상 모델 참조
  - 추론 가능한 경우 역참조 관계 메타데이터
  - 추론 가능한 경우 메서드 단위의 연관 모델 참조
- 해결되지 않거나 부분적으로만 해결된 참조는 조용히 버리지 말고 구조화된 진단 정보로 남겨야 한다.
- 사용자 정의 메서드는 프레임워크에서 상속된 메서드가 아니라 해당 모델에 직접 선언된 메서드를 뜻한다.
- `@property`가 붙은 멤버는 실제 DB 필드와 구분해서 표현해야 하며, UI에서 저장 필드와 계산 속성을 명확히 분리할 수 있어야 한다.
- Django `choices`를 사용하는 필드는 가능한 경우 enum 성격의 메타데이터와 정적으로 추론 가능한 옵션 목록을 노출해야 한다.

### 4.3 관계 그래프 구성

- 시스템은 프로젝트 전체 모델 관계를 표현하는 그래프를 구성해야 한다.
- 그래프 빌더는 DFS 또는 이에 준하는 순회 전략으로 관계를 따라가야 한다.
- 그래프에는 다음이 포함되어야 한다.
  - `ForeignKey`
  - `OneToOneField`
  - `ManyToManyField`
  - 역참조 관계
  - 관계가 전혀 없는 독립 모델
- 최종 그래프는 하나의 시작 모델에서 연결된 컴포넌트만이 아니라, 프로젝트에서 파악 가능한 전체 구조를 표현해야 한다.

### 4.4 ERD 렌더링

- ERD는 벡터 형식으로 렌더링되어야 한다.
- 더 강한 대안이 정당화되지 않는 한 SVG를 기본 렌더링 타깃으로 한다.
- ERD는 VS Code 웹뷰 안에서 표시되어야 한다.
- 각 모델은 ERD 관례를 따르는 테이블 형태의 노드로 렌더링되어야 한다.
- 관계선은 관련 테이블 사이에 자동으로 연결되어야 한다.
- Django `choices`가 선언된 필드는 enum 성격의 필드임을 시각적으로 식별할 수 있어야 한다.
- choice 메타데이터가 있으면 해당 필드의 선택 가능한 옵션을 UI에서 표시할 수 있어야 한다.
- 각 모델 뷰는 실제 모델 필드와 별도로 `@property`가 붙은 계산 속성을 표시할 수 있어야 한다.
- 테이블을 선택하면 해당 모델의 사용자 정의 메서드를 볼 수 있어야 한다.
- 메서드가 표시될 때, 추론된 연관 모델 참조를 바탕으로 해당 메서드와 관련된 테이블들을 하이라이트할 수 있어야 한다.

### 4.5 상호작용

- 사용자는 ERD 뷰에서 테이블을 직접 이동시킬 수 있어야 한다.
- 사용자는 선택한 테이블을 숨길 수 있어야 한다.
- 사용자는 테이블을 선택해 해당 모델의 로직 관련 메타데이터를 확인할 수 있어야 한다.
- 사용자는 테이블별로 메서드 표시를 토글할 수 있어야 한다.
- 사용자는 테이블별로 `@property` 계산 속성 표시를 토글할 수 있어야 한다.
- 사용자는 테이블별로 메서드 기반 연관 테이블 하이라이트를 켜고 끌 수 있어야 한다.
- UI는 향후 뷰 조작 기능을 큰 구조 변경 없이 추가할 수 있어야 한다.
- 상호작용 계층은 필터링, 고정, 접기, 그룹화 같은 기능을 이후에 무리 없이 넣을 수 있도록 설계되어야 한다.

### 4.6 레이아웃 옵션

- 사용자는 테이블 정렬 방식을 선택할 수 있어야 한다.
- 초기 레이아웃 옵션에는 최소한 다음이 포함되어야 한다.
  - 계층형
  - 원형
  - 클러스터형
- 레이아웃 적용 로직은 렌더링 로직과 분리되어야 하며, 새로운 전략을 나중에 쉽게 추가할 수 있어야 한다.

### 4.7 관계선 라우팅과 가독성

- 관계선은 표준 ERD 의미 체계를 따라야 한다.
- 라우팅 시스템은 테이블을 자동 연결하면서 가능한 한 선 겹침을 줄여야 한다.
- 두 선이 교차할 때 사용자가 병합으로 오해하지 않도록, 교차 여부를 시각적으로 구분해야 한다.
- 자동 배치는 작은 예시뿐 아니라 중대형 스키마에서도 가독성을 우선해야 한다.

## 5. UI 및 UX 요구사항

- 주요 시각화 영역은 VS Code 내부 웹뷰여야 한다.
- 연결된 모델 그룹과 분리된 모델 그룹 모두 읽기 쉬워야 한다.
- UI는 저장되는 모델 필드, enum 성격의 choice 필드, `@property` 계산 속성, 사용자 정의 메서드를 명확히 구분해서 보여줘야 한다.
- 메서드 기반 테이블 하이라이트는 구조적 ERD 관계선과 혼동되지 않는 보조 이해 수단으로 동작해야 한다.
- UI 구조는 시각 스타일, 노드 렌더링, 엣지 렌더링을 서로 독립적으로 발전시킬 수 있을 정도로 모듈화되어야 한다.
- 확장은 상태 관리, 레이아웃 계산, 그래프 데이터, 프레젠테이션 관심사를 분리해야 한다.

## 6. 아키텍처 요구사항

### 6.1 계층 구조

프로젝트는 최소한 다음과 같은 계층으로 명확히 구성되어야 한다.

- VS Code extension host 계층
- 분석 오케스트레이션 계층
- Rust 분석기 계층
- 그래프 및 도메인 계층
- 레이아웃 계층
- 렌더링 및 웹뷰 UI 계층

각 계층은 다른 계층의 구체 구현보다는 안정적인 인터페이스에 의존해야 한다.

### 6.2 확장성

- 코드베이스는 추상화와 유지보수 가능한 경계를 우선해야 한다.
- 렌더링, 레이아웃, 분석 동작은 명확한 인터페이스를 통해 교체하거나 확장할 수 있어야 한다.
- UI 전용 로직은 Rust 분석기 내부 구현과 강하게 결합되면 안 된다.
- 그래프 및 도메인 타입은 다른 렌더러나 향후 export 포맷에서도 재사용 가능해야 한다.

### 6.3 파일 크기 제약

- 어떤 단일 소스 코드 파일도 500줄을 넘기면 안 된다.
- Markdown 문서는 이 제한의 예외로 둘 수 있으며, 3000줄 이하를 기준으로 한다.
- 소스 모듈이 코드 파일 한계에 가까워지면 단일 파일로 유지하지 말고 책임 단위로 분리해야 한다.

## 7. 품질 요구사항

- 정적 분석은 실제 Django 프로젝트에서도 반응성이 느껴질 만큼 빨라야 한다.
- 부분 분석 실패가 발생해도 가능한 한 정상 동작을 유지하고, 진단 정보를 노출해야 한다.
- 렌더러는 확대와 상호작용 중에도 벡터 품질을 유지해야 한다.
- 프로젝트 구조는 장기적인 성장에도 대규모 재작성 없이 확장 가능해야 한다.

## 8. 권장 내부 모듈 방향

구현은 대체로 다음과 같은 책임 분리를 갖는 모듈 구성을 지향하는 것이 좋다.

- 확장 명령과 activation
- 워크스페이스 및 프로젝트 탐색
- Rust 분석기 브리지
- 모델 메타데이터 스키마
- 관계 그래프 빌더
- 레이아웃 전략 레지스트리
- 엣지 라우팅 엔진
- 웹뷰 상태 저장소
- ERD 렌더러
- 사용자 상호작용 컨트롤러

이 목록은 방향 제시이며 강제 사항은 아니지만, 최종 구조는 이와 동등한 수준의 관심사 분리를 유지해야 한다.

## 9. 인수 기준

첫 번째로 usable한 버전은 다음을 만족해야 한다.

- 사용자가 VS Code에서 Django 프로젝트를 열고 ERD 뷰를 실행할 수 있다.
- 확장이 정적 분석을 통해 여러 앱의 모델을 찾는다.
- 결과 그래프에 관계로 연결된 모델과 고립된 모델이 모두 포함된다.
- ERD가 벡터 기반 웹뷰 렌더러로 표시된다.
- 사용자가 테이블을 이동하고 숨길 수 있다.
- 사용자가 테이블을 선택해서 사용자 정의 모델 메서드를 볼 수 있다.
- 사용자가 모델별 `@property` 계산 속성을 볼 수 있다.
- 사용자가 Django `choices` 필드를 enum 성격의 필드로 인식하고, 가능한 경우 선택 옵션 목록을 볼 수 있다.
- 사용자가 테이블별로 메서드 표시, `@property` 표시, 메서드 기반 테이블 하이라이트를 켜고 끌 수 있다.
- 사용자가 계층형, 원형, 클러스터형 레이아웃으로 전환할 수 있다.
- 관계선이 ERD 의미 체계를 따라 자동으로 그려진다.
- 엣지 라우팅이 선 겹침을 줄이려 시도하고, 교차를 시각적으로 표시한다.
- 메서드 선택 또는 조회 시 해당 메서드와 연관된 테이블이 하이라이트된다.
- 코드베이스가 모듈식이고 계층화되어 있으며, 과도하게 큰 파일이 없다.

## 10. 미래 확장 방향

초기 구현은 UI나 분석 파이프라인을 좁은 방식으로 고정하지 않아야 한다. 이 명세는 인터페이스와 분리 가능한 모듈을 우선하며, 이후 export, 검색, 필터링, 레이아웃 저장, 서브그래프 그룹화, 더 정교한 Django 관계 추론, 더 깊은 메서드 단위 의존성 분석 같은 기능을 큰 재설계 없이 추가할 수 있어야 한다.
